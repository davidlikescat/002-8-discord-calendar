# PRD 생성 프롬프트
## 디스코드 일정 정리 → 구글 캘린더 자동 등록 에이전트

> 이 문서는 PRD를 작성하기 위한 **프롬프트(지시문)**다.
> 이 내용을 Claude에게 입력하면 ① 요구사항 정의서 → ② PRD 순서로 산출물을 생성한다.

---

## 1. 당신의 역할

당신은 시니어 프로덕트 매니저이자 솔루션 아키텍트다.
1인 개발자가 Python으로 빠르게 구현할 수 있는, 구체적이고 실행 가능한 문서를 작성한다.
추상적 미사여구를 배제하고, 개발자가 바로 설계·코딩에 착수할 수 있는 수준의 디테일을 담는다.

## 2. 만들 제품

### 한 줄 정의
사용자가 디스코드에 올린 일정 정보(텍스트·카카오톡 캡처·사진)를 자동으로 인식·구조화하여
구글 캘린더에 등록하는 개인용 에이전트.

### 핵심 사용자 흐름
1. 사용자가 외부에서 일정 정보를 수신한다 (문자, 카카오톡 대화, 안내문 사진 등).
2. 사용자가 그 정보를 디스코드 특정 채널에 올린다 — 텍스트 붙여넣기 또는 이미지 첨부.
3. 디스코드 봇이 새 메시지를 감지한다.
4. 디스코드 봇이 입력(텍스트·이미지)을 Claude CLI에 전달한다.
5. Claude CLI가 이미지면 글자를 직접 읽고(OCR), 텍스트면 그대로 받아 일정을 구조화 추출한다 — 날짜/시간/제목/장소/설명. 한 입력에 여러 일정이 있을 수 있다.
6. 봇이 추출 결과를 사용자에게 디스코드로 보여주고 확인을 요청한다.
7. 사용자가 승인하면 구글 캘린더에 등록한다.
8. 등록 결과(성공/실패/링크)를 디스코드로 회신한다.

### 입력 유형
- A. 일반 텍스트 메시지 (붙여넣기)
- B. 카카오톡 대화 캡처 이미지
- C. 일정 안내문·포스터·손글씨 등 사진

### 출력
- 구글 캘린더 이벤트 (제목, 시작/종료 일시, 장소, 설명)

## 3. 알려진 컨텍스트 / 제약

- **개발자**: 1인 개발자. 빠른 MVP 우선. 개인용 도구(상용 서비스 아님).
- **핵심 구현 방식 (확정)**: **Claude CLI 기반**으로 구현한다.
  - 이미지 속 글자 인식(OCR)과 일정 구조화 추출을 모두 Claude CLI의 비전·추론 능력으로 처리한다 — 별도 OCR 엔진을 두지 않는 것을 1순위로 검토.
  - 디스코드 봇(Python)이 입력을 받아 Claude CLI를 headless 모드로 호출하고, 그 결과를 받아 처리하는 구조.
  - 캘린더 등록은 ① Claude CLI가 MCP(Google Calendar MCP)로 직접 수행 ② Python이 Google Calendar API를 직접 호출 — 두 방식을 비교 검토.
- **기술 스택**: Python (디스코드 봇 = discord.py) + Claude CLI(핵심 엔진).
- **기존 자산**: 디스코드 기반 자동화 프로젝트를 이미 여러 개 보유 —
  `003_discord_to_notion`, `002-4_discord_news_notion`.
  디스코드 봇 구조·인증 패턴을 재사용할 수 있다. PRD 작성 시 이 자산의 재사용 가능성을 검토할 것.
- **계정**: 구글 캘린더는 davidlikessangria@gmail.com 계정 기준.
- **언어**: 입력 일정 정보는 대부분 한국어. 타임존은 KST(Asia/Seoul).

## 4. 산출물 (2개, 순서대로)

### 4-1. 요구사항 정의서
- 기능 요구사항(FR)·비기능 요구사항(NFR)을 ID 부여하여 표로 정리
- 입력 유형(A/B/C)별 처리 요구사항
- 우선순위(MoSCoW: Must / Should / Could / Won't) 표기

### 4-2. PRD
- 요구사항 정의서를 바탕으로 한 제품 사양 문서
- 아래 7장의 목차를 따른다

## 5. 작성 절차 (반드시 이 순서로)

1. **STEP 1 — 질문**: 아래 8장의 핵심 설계 질문 + 추가로 발견한 불명확 사항을 정리해
   사용자에게 먼저 묻는다. 추측으로 진행하지 않는다.
2. **STEP 2 — 요구사항 정의서**: 사용자 답변을 반영해 요구사항 정의서를 작성하고 검토받는다.
3. **STEP 3 — PRD**: 확정된 요구사항을 바탕으로 PRD를 작성한다.

## 6. 요구사항 정의서가 다뤄야 할 항목

- [ ] 입력 유형(A/B/C)별 인식 요구사항과 한계
- [ ] OCR 정확도 요구 수준, 실패 시 처리
- [ ] 일정 추출 항목 정의: 제목 / 시작 / 종료 / 장소 / 설명 / 참석자 — 필수 vs 선택
- [ ] 상대 날짜·모호 표현 처리 ("다음 주 화", "담주", 연도 없는 "3/14")
- [ ] 시간 미지정 시 종일 일정 처리 규칙
- [ ] 한 입력에 여러 일정이 있을 때 처리
- [ ] 추출 실패·모호 시 사용자에게 되묻는 흐름
- [ ] 캘린더 등록 전 사용자 확인(승인 / 수정 / 취소) 흐름
- [ ] 중복 일정 감지·방지
- [ ] 비기능 요구사항: 응답 시간, API 비용, 보안(토큰 보관), 로깅·에러 처리

## 7. PRD 목차 (7장)

1. **개요** — 배경, 문제 정의, 목표, 성공 지표
2. **사용자 & 시나리오** — 페르소나(=개발자 본인), 핵심 유스케이스, 엣지케이스
3. **기능 사양** — 기능별 상세(입력→처리→출력), 디스코드 UX(명령 / 리액션 / 임베드),
   일정 추출 JSON 스키마
4. **시스템 아키텍처** — 컴포넌트 다이어그램, 데이터 흐름,
   외부 연동(디스코드 API, OCR, LLM, Google Calendar API), 인증 방식
5. **기술 결정** — Claude CLI 호출 방식(headless 실행·프롬프트 설계·입출력 포맷),
   이미지 인식 정확도 및 OCR 폴백 여부, 캘린더 등록 경로(MCP vs API),
   기존 디스코드 자산 재사용 범위
6. **개발 계획** — MVP 범위, 마일스톤, 디렉터리 구조, 환경변수 목록
7. **리스크 & 향후 과제** — 인식 정확도, API 비용, 확장 아이디어

## 8. 반드시 결정해야 할 핵심 설계 질문

PRD 작성 전 사용자에게 다음을 질문하라:

1. **디스코드 트리거 방식** — 특정 채널 전체 감시 vs 슬래시 명령(`/일정`) vs 봇 멘션. 어느 것?
2. **확인 단계** — 캘린더 등록 전 항상 사용자 승인을 받을지, 신뢰도 높으면 자동 등록할지?
3. **승인 UX** — 디스코드 버튼 / 이모지 리액션(✅ ❌) / 텍스트 답장 중 무엇으로 승인·수정?
4. **OCR 정확도 / 폴백** — 이미지 인식은 Claude CLI 비전으로 처리하는 것이 기본.
   카카오톡 캡처·손글씨 사진에서 정확도가 부족할 경우 별도 OCR 엔진(Google Vision, Naver CLOVA 등) 폴백을 둘지?
5. **캘린더 등록 경로** — Claude CLI가 MCP로 직접 등록 vs Python이 Google Calendar API 호출 — 어느 쪽 선호?
6. **대상 캘린더** — 기본 캘린더에 등록 vs 전용 캘린더 신설? 일정 종류별 분리 필요?
7. **반복 일정** — "매주 회의" 같은 반복 일정 지원 범위 (MVP 포함 여부)?
8. **수정 흐름** — 추출 결과가 틀렸을 때 사용자가 디스코드에서 직접 고치게 할지,
   원문을 다시 올리게 할지?
9. **알림·리마인더** — 캘린더 기본 알림 외 별도 리마인더가 필요한지?
10. **운영 환경** — 로컬 상시 실행 vs 클라우드 배포? (기존 프로젝트 방식 참고)

## 9. 작성 품질 기준

- 한국어, 간결체. 표·목록 적극 활용.
- 모든 요구사항·기능에 ID 부여 (FR-01, NFR-01 등).
- "잘 처리한다" 같은 모호한 표현 금지 — 측정 가능·검증 가능하게 작성.
- 1인 개발자가 감당 가능한 MVP를 1차 범위로 명확히 분리할 것.
- 결정하지 못한 사항은 "미정"으로 표기하고, 옵션과 트레이드오프를 함께 제시.
