## 4. 핵심 설계 결정

### 4-1. Claude CLI 호출 (요구사항 §8-1)
- 형태: `claude --print --dangerously-skip-permissions <prompt>` (asyncio subprocess)
- 입력: 프롬프트 문자열(텍스트 + 첨부 이미지 절대경로 포함)
- 출력: stdout. JSON만 반환하도록 프롬프트로 강하게 지시 + `parse_json_loose`로 코드펜스·잡설 견딤
- 타임아웃: 추출 120s / 등록 120s (.env 조절)
- 비전: 첨부를 봇이 `data/attachments/<msg_id>/`에 미리 다운로드, 경로를 프롬프트에 넣고 **Read 도구로 읽으라고 명시**

### 4-2. Calendar MCP 호출 (요구사항 §8-2)
- 봇은 직접 MCP를 호출하지 않는다. **Claude CLI가 MCP 도구를 호출**하도록 프롬프트로 지시.
- 현재 사용자 환경: `claude.ai Google Calendar` MCP가 `claude mcp list`에서 ✓ 상태.
- 중복 판정: 등록 프롬프트에서 list/search로 동일 시간대(±5분)·동일/유사 제목 체크 후 insert.

### 4-3. 캐치업 (요구사항 §8-6)
- 마지막 처리 시점 저장: SQLite `meta.last_processed_message_id` (snowflake 그대로).
- 범위: 커서 있으면 그 이후, 없으면 최근 `CATCHUP_MAX_HOURS` 시간.
- 한 번에 최대 200건. 봇 메시지·이미 처리한 메시지(`_processing`)는 제외.

### 4-4. 수정 UX (요구사항 §8-5)
- 두 경로 모두 지원:
  - **[수정] 버튼 → 모달** (`EditModal`) — 자연어 한 줄 입력
  - **"수정: ..." 답장** — 결과 임베드에 reply
- 둘 다 `update_event`로 모이며, Claude CLI가 events.get → 변경 해석 → events.patch.

### 4-5. 신뢰도 임계값 (요구사항 §8-4)
- `CONFIDENCE_THRESHOLD = 0.6` (기본). `.env`로 튜닝.
- 임계값 미만이어도 자동 등록(FR-23) 하되 임베드를 주황색 + "⚠️" 헤더로.

## 7. 셀프 리뷰 (이슈 + 수정)

### 🐛 R-01 [High] EditModal이 binding을 못 찾는 버그
- **증상**: 모달 `on_submit`에서 `interaction.message`는 None이거나 모달과 무관한 메시지. 따라서 `embed_id`를 알 수 없어 binding 조회 실패.
- **원인**: discord.py에서 modal interaction은 component interaction과 달리 원본 메시지를 자동 참조하지 않음.
- **수정**: 버튼 핸들러에서 binding을 먼저 조회해 modal 인스턴스에 `event_id`·`title`을 전달.

### 🐛 R-02 [Medium] 이미지 OCR 지시가 약함
- **증상**: 프롬프트에 이미지 경로만 나열하면 Claude가 Read 도구로 읽지 않고 추측할 수 있음.
- **수정**: 프롬프트에 "Read 도구로 이 파일을 직접 읽고 비전 인식하라" 명시.

### 🐛 R-03 [Medium] 캐치업이 사용자 메시지를 못 보고 처리할 수 있음
- **증상**: `on_ready`에서 `_catchup` 태스크가 떨어진 직후, `on_message`로 신규 메시지가 들어오면 두 경로가 동시에 처리할 가능성.
- **현재 방어**: `_processing` set으로 같은 message.id 중복 방지.
- **판정**: OK. 추가 수정 불필요.

### 🐛 R-04 [Low] "수정:" 답장이 봇 임베드가 아닌 메시지에 달리면?
- **증상**: 사용자가 자기 메시지에 reply하며 "수정: ..."로 시작하면 신규 처리로 흘러감.
- **현재 동작**: `_maybe_handle_edit_reply`가 binding 없으면 False 반환 → 신규 처리. 사용자가 잠재적으로 캘린더 일정 의도로 적은 메시지가 새 일정으로 들어갈 수 있음.
- **판정**: 사용자 책임 범위. Claude가 "수정: ..." 텍스트에서 일정을 못 찾으면 events=[]로 떨어지므로 큰 부작용 없음.

### 🐛 R-05 [Low] launchd plist의 PATH에 `node` 경로 누락
- **증상**: Claude CLI가 npm 실행 환경 의존. `node`/`npx` 위치가 PATH에 있어야 MCP 동작.
- **수정**: plist `EnvironmentVariables.PATH`에 일반 npm 경로 보강.

### ✅ R-06 [확인됨] 중복 처리 위험
- `state.set_last_message_id`를 처리 완료 후 호출. 처리 중 봇이 죽으면 다음 캐치업이 같은 메시지를 다시 처리할 수 있음.
- **판정**: 등록 프롬프트의 중복 판정(FR-16)으로 캘린더 중복은 막힘. 봇 응답이 두 번 갈 수는 있으나 수용 가능.

### ✅ R-07 [확인됨] Python 3.9 호환성
- `zoneinfo`는 3.9에 추가됨 → 사용 환경(3.9.6)에서 OK.
- 단, 타입 힌트 `dict | None` 같은 PEP 604 문법은 `from __future__ import annotations`로 회피됨 → 모든 모듈 상단에 추가.

## 8. 검증 계획

1. **단위**: `parse_json_loose` 코드펜스/잡설 케이스 (이미 통과).
2. **모듈 import 스모크**: `python -c "import bot"` (이미 통과).
3. **수동 통합**:
   - 텍스트 일정 1건 → 임베드 회신 확인
   - 다중 일정 → 임베드 2개 회신 확인
   - 이미지(카톡 캡처) → 되묻기 임베드 확인
   - 결과 임베드 [수정] 버튼 → 모달 → 갱신 확인
   - "수정: ..." 답장 → 갱신 확인
   - [삭제] 버튼 → 캘린더에서 삭제 확인
   - 봇 강제 종료 후 재시작 → 캐치업 동작 확인
